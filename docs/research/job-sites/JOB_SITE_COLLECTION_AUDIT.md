# 채용 사이트 수집 가능 필드 감사

확인일: 2026-05-13

## 목적

크롤러 구현 전에 각 채용 사이트에서 브라우저 자동화 없이 확보 가능한 정보를 먼저 확인한다. 포트폴리오 데모 목적상 실시간 수집은 목표가 아니며, 1시간 단위 배치 수집으로 확장 가능한 구조를 우선한다.

## 공통 판단 기준

| 항목 | 확인 내용 |
| --- | --- |
| 목록 접근 | 로그인 없이 채용 목록 HTML이 보이는지 |
| 상세 접근 | 로그인 없이 상세 공고 HTML이 보이는지 |
| 목록 필드 | 제목, 회사명, 지역, 경력, 학력, 마감일 등 목록에서 바로 얻을 수 있는 값 |
| 상세 필드 | 자격요건, 우대사항, 기술스택, 고용형태, 근무조건, 기업정보 등 상세에서 얻을 수 있는 값 |
| 수집 방식 | HTTP + HTML 파싱, 숨은 JSON/API 조사, Playwright fallback 필요 여부 |
| 리스크 | 로그인, 캡차, JS 의존, 크롤링 금지 문구, 차단 가능성 |

## 판단 등급 기준

| 등급 | 의미 |
| --- | --- |
| 우선 구현 | 로그인 없이 목록/상세 데이터가 보이고, 현재 `JobPosting` 구성에 필요한 필드가 충분함 |
| 조건부 구현 | 데이터는 얻을 수 있지만 카테고리 필터, 중복 제거, 캡차 가능성 등 추가 방어가 필요함 |
| 후순위 | 일부 데이터는 보이지만 로그인, JS 의존, 봇 검증, 상세 접근 불안정성이 큼 |
| 제외 또는 보류 | 수집 금지 문구가 명확하거나 공개 HTML만으로 공고 데이터를 얻기 어려움 |

## 추천 구현 순서

1. 사람인: 상세 필드가 가장 풍부하고 데모 가치가 높음
2. 잡코리아: 목록 수집만으로도 데이터량과 필드가 충분함
3. 캐치: 상세 HTML에서 포트폴리오에 좋은 기업/직무 정보 확보 가능
4. 링커리어: 신입/인턴 카테고리로 제한해서 구현
5. Indeed Korea: 모바일 검색 결과 기준으로 실험

후순위 또는 보류:

- LinkedIn Jobs Korea: 로그인/마감공고/차단 리스크
- 잡플래닛: 채용 목록 데이터 노출 약함
- 로켓펀치: JS/봇 검증
- 프로그래머스 채용: 크롤링/스크래핑 거부 문구

## 크롤러 설계 메모

- `source`: 사이트 키
- `sourceJobId`: 사이트 내 공고 ID 또는 상세 URL hash
- `title`: 공고명
- `company`: 회사명
- `location`: 근무지역
- `careerLevel`: 경력 조건
- `education`: 학력 조건
- `employmentType`: 고용형태
- `salary`: 급여
- `deadline`: 마감일
- `skills`: 기술/직무 키워드
- `description`: 상세 설명 또는 요약
- `sourceUrl`: 원본 URL
- `raw`: 원본 HTML/JSON 일부 저장용

현재 앱의 `JobPosting` 타입은 필드가 작으므로, 크롤러 내부 정규화 타입은 더 넓게 잡고 API 응답에서 필요한 필드만 내려주는 방식이 낫다.
